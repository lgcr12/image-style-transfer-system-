# 复试 / 答辩高频问答

本文档用于复试、毕业答辩或项目展示时快速准备讲解。回答应尽量围绕“需求 - 设计 - 实现 - 测试 - 优化”展开。

## 1. 你的项目整体是做什么的？

本项目是一个基于深度学习的图像风格迁移与智能重绘系统。用户上传图片后，可以选择传统风格迁移模型或 Stable Diffusion / LoRA 模型进行图像重绘。系统提供 Web 页面、任务队列、模型管理、本地历史记录和云端 ComfyUI GPU 推理能力。

## 2. 系统整体架构是什么？

系统采用 Browser/Server 架构。前端负责图片上传、参数配置、任务提交、状态轮询和结果展示；后端使用 FastAPI 提供接口；任务队列负责执行耗时推理；SQLite 保存任务状态；文件系统保存上传图、结果图和模型文件；本地模型负责传统风格迁移和部分 SD 推理；云端 ComfyUI 用于高算力推理。

## 3. 为什么使用 FastAPI？

FastAPI 适合 Python AI 项目，接口定义清晰，支持异步能力、文件上传、JSON 返回和自动文档扩展。图像生成模型主要使用 Python 生态，因此 FastAPI 可以更方便地与 PyTorch、Diffusers、ONNX Runtime 和 ComfyUI 调用逻辑结合。

## 4. 为什么需要任务队列？

图片生成是耗时任务，如果直接在 HTTP 请求中同步执行，会导致请求长时间阻塞，前端体验差，也容易超时。任务队列可以先返回 `job_id`，后台慢慢执行模型推理，前端通过状态接口轮询进度。这样既提升了稳定性，也方便扩展暂停、取消、批量任务等功能。

## 5. 任务状态是怎么流转的？

任务创建后进入 `PENDING/queued`，Worker 取出任务后变为 `RUNNING/running`，生成成功后变为 `SUCCESS/done`，异常时变为 `FAILED/error`，用户取消时变为 `CANCELLED`。状态、进度和错误原因会写入任务记录，前端通过 `/api/status/{job_id}` 获取。

## 6. SQLite 为什么够用？

本项目是单机或小规模 Web 应用，数据主要是任务索引、文件路径、状态和参数 JSON，不涉及高并发复杂事务。SQLite 部署简单、无需独立数据库服务，适合毕业设计阶段。后续如果多用户并发量上升，可以迁移到 MySQL 或 PostgreSQL。

## 7. 图片为什么不直接存数据库？

图片和模型文件体积较大，直接写入数据库会增加数据库体积和读写压力。系统采用“数据库保存路径 + 文件系统保存文件”的方式，更简单也更适合大文件管理。

## 8. SD 和传统风格迁移有什么区别？

传统风格迁移主要迁移颜色、纹理和笔触，速度较快，但语义控制能力有限。Stable Diffusion 是扩散生成模型，可以结合提示词、LoRA 和基础模型进行语义级重绘，生成能力更强，但速度更慢、资源消耗更高。

## 9. LoRA 是什么？

LoRA 是一种低秩适配微调方法。它不需要完整训练大模型，只保存较小的增量权重，用于控制特定画风、角色或视觉效果。系统中 LoRA 作为风格模型，与 SD1.5 或 SDXL 基础模型组合使用。

## 10. 为什么 LoRA 要绑定基础模型？

不同 LoRA 通常基于不同模型体系训练，例如 SD1.5 LoRA 不能直接用于 SDXL，SDXL LoRA 也不适合 SD1.5。系统通过配置文件记录 LoRA 推荐基础模型，用户选择风格后自动匹配，减少手动切换和出错概率。

## 11. 云端推理是怎么实现的？

系统通过 SSH 连接远程 GPU 服务器，将输入图上传到 ComfyUI 的 input 目录，构造 ComfyUI workflow JSON，通过 ComfyUI API 提交任务，等待结果生成后再把 output 目录中的结果图下载回本地。

## 12. 为什么要接入云端 ComfyUI？

SDXL 和大模型对显存要求较高，本地机器可能无法流畅运行。接入云端 ComfyUI 后，可以使用 RTX 4090 等云端 GPU 执行推理，本地只负责 Web 服务和结果展示。

## 13. 如果云端没有 GPU 会怎样？

系统会检测 `nvidia-smi`。如果返回 `No devices were found`，说明实例处于无卡模式或 GPU 不可用。此时 ComfyUI 无法正常启动，系统会返回“云端 GPU 不可用”的诊断提示。

## 14. 如果 ComfyUI 返回 HTTP 400 怎么处理？

HTTP 400 通常是工作流校验失败，常见原因是 checkpoint 或 LoRA 文件名不在 ComfyUI 可用列表中，或者模型体系不兼容。系统会提示检查云端模型映射，并建议刷新模型列表或改用 ComfyUI 实际识别到的文件名。

## 15. 系统如何处理异常？

后端会捕获模型文件不存在、GPU 不可用、显存不足、ComfyUI 连接失败、SSH 上传失败、工作流校验失败等异常，并将诊断信息写入任务状态。前端轮询状态时展示错误标题和建议，而不是只显示未知错误。

## 16. 多用户同时生成怎么办？

当前系统通过任务队列控制生成任务串行或限制并发，避免多个 SD 任务同时占用显存。后续可以扩展为多 Worker、多 GPU 调度，或者按用户维度建立任务队列。

## 17. 项目有哪些数据表？

核心是 `jobs` 表，用于保存 `job_id`、任务类型、状态、进度、输入路径、输出路径、参数 JSON、错误信息、创建时间和更新时间。还有 `gallery_items` 用于保存发布到画廊或历史展示的记录。

## 18. 这个项目的技术亮点是什么？

主要亮点包括：传统风格迁移与 SD 重绘融合、LoRA 与基础模型自动绑定、异步任务队列、云端 ComfyUI GPU 接入、本地历史记录、模型导入管理、错误诊断和参数推荐。

## 19. 项目还有什么可以优化？

可以继续优化多用户权限、自动化测试、模型兼容性校验、云端实例自动开关、结果质量评分、批量任务、WebSocket 实时进度、Docker 部署和 CI/CD。

## 20. 如果让你重新设计，你会改哪里？

我会把模型管理、任务调度和云端推理抽成更清晰的服务模块；用 WebSocket 替代前端轮询；数据库从 SQLite 迁移到 PostgreSQL；任务队列使用 Redis Queue 或 Celery；并增加 Docker Compose 便于部署。
